This note answers the narrower practical question:

> If I spawn a Pi agent with the SCM extension enabled, is the original intended behavior actually there yet?

It is intentionally capability-focused, not code-plan-focused.

## Short answer

As of 2026-03-11T22:26:51Z:

- The **versioned storage/runtime substrate is real**.
- The **explicit agent-facing self-context-management interface is not finished**.
- Therefore the project is **not yet at the full intended state** of: “spawn a Pi agent and it can deliberately manage its own context as a normal operating mode.”

More bluntly:
- **versioned store:** yes
- **metadata layer:** yes
- **session-scoped objects:** yes
- **internal active/pinned/inactive model:** yes
- **automatic file/tool observation:** yes, for a wired subset
- **agent can explicitly curate its own context:** **no, not yet**
- **SCM is clearly/fully authoritative over final Pi context:** **not yet demonstrated clearly enough**

## What is actually implemented now

### 1) Versioned context store exists

Implemented.

Current shape:
- Default backend is SQLite through `StoragePort`.
- Default DB path in the current workspace is `.pi/self-context-manager.sqlite`.
- Sessions are separated inside that store by object identity rather than by one DB per chat.

Current object families include:
- `session:<sessionId>`
- `chat:<sessionId>`
- `system_prompt:<sessionId>`
- `file:<absolutePath>`
- toolcall metadata objects

Interpretation:
- The current design is **shared SQLite store + separate session/chat/system objects per Pi session**.
- It is **not** “new SQLite DB per chat by default.”

### 2) Context state is versioned, not just mutable in-memory state

Implemented.

At the runtime/storage level, the following are represented through immutable version writes:
- session state
- chat object
- system prompt object
- file objects when indexed/updated

This part appears materially implemented, not merely stubbed.

### 3) Metadata layer exists

Implemented.

The runtime maintains a metadata pool that includes at least:
- file metadata
- toolcall metadata

Metadata is not just stored internally; it is also surfaced into assembled context.

### 4) Internal context sets exist

Implemented internally.

The runtime model includes:
- `active_set`
- `inactive_set`
- `pinned_set`

The runtime also exposes internal methods for:
- `activate(id)`
- `deactivate(id)`
- `pin(id)`
- `unpin(id)`

Important qualifier:
- These capabilities exist **inside the runtime**, but that does **not** mean the agent currently has a finished way to use them.

### 5) Automatic observation of some tool activity exists

Implemented for a wired subset.

The current wrapper updates SCM state when these tool paths occur:
- `read`
- `write`
- `edit`
- `ls`
- `find`
- `grep`
- `bash`

Interpretation:
- SCM can build/update managed state from ordinary tool use.
- This is **not** the same as “all Pi tools are integrated.”

### 6) File-watch/update path exists

Implemented path exists.

The runtime sets up a `chokidar` watcher and has handlers for:
- add
- change
- unlink

Important qualifier:
- This is enough to say that live-sync behavior is implemented in principle for tracked files.
- It is **not yet enough** to claim, without qualification, that every discovered object can be relied on to stay synchronized in all normal usage patterns.

## What is not finished yet

### 1) Explicit agent self-context editing is not finished

This is the main missing capability.

The runtime has internal operations for context editing, but the current agent-facing wrapper exposes only:
- `/scm-status`
- `/scm-read`
- `/scm-dump`

Missing from the current agent-facing surface:
- explicit activate
- explicit deactivate
- explicit pin
- explicit unpin
- practical browse/list/search/show operations over known managed objects

Meaning:
- The agent can currently change managed state **indirectly** through reads/writes/tool observation.
- The agent cannot yet **deliberately curate its own context** through a finished first-class interface.

This is the biggest gap between the current repo and the original intended behavior.

### 2) Discoverability/introspection is not finished enough

Not finished.

The agent still lacks a clear practical interface for answering questions like:
- what is currently active?
- what is pinned?
- what objects are known?
- what is metadata-only vs content-loaded?
- what object should be activated next?

The runtime has the raw state; the usable inspection surface is still missing.

### 3) Authoritative context ownership is not clearly complete

Not yet clear/finished enough.

The current runtime clearly assembles context and transforms Pi messages, but the repo does not yet clearly establish, in a way that is easy to rely on operationally, that:

> SCM is the fully authoritative owner of final agent context rather than a structured augmentation layer around Pi’s normal context flow.

This is important because the original intended behavior is the stronger version:
- SCM decides the real working context
- the agent edits that context deliberately

That stronger claim should not be treated as complete yet.

### 4) Agent operating protocol is not finished

Not finished.

Even once the controls exist, the model needs a normal operating contract for when to:
- read/index something new
- keep an item as metadata-only
- activate it into working context
- pin it as a durable anchor
- demote it later

That policy is not yet established as a finished agentic interface.

## Capability-by-capability assessment

### “If I turn on the extension, does it just work?”

Answer depends on the bar.

Lower bar:
- versioned storage starts
- session/chat/system/file state exists
- metadata appears
- some tool/file observation occurs

For that lower bar: **mostly yes**.

Original intended bar:
- the agent can explicitly manage its own context as part of normal operation
- SCM is the effective context authority
- the agent can inspect/change/pin/unpin/focus its context intentionally

For that intended bar: **no, not yet**.

### “Can the agent context-edit?”

Current answer: **not in the full intended sense**.

The internal runtime can do it.
The current agent-facing interface does not yet expose that as a finished capability.

### “Is session state separated per chat?”

Yes, via object/session identity in a shared store.

Current default is:
- one workspace DB (`.pi/self-context-manager.sqlite`)
- separate `session/chat/system_prompt` objects per session

That is a valid design, but it should be understood as **shared store, session-scoped objects**, not per-chat DB files.

### “When files change, does SCM update?”

Best current claim:
- live update path is implemented for tracked files
- this should not yet be advertised as a fully proven universal capability across all discovered object states without more explicit validation and clearer interface guarantees

## The practical delta still remaining

To reach the intended state, the missing capabilities are not “more storage.”
They are:

1. **explicit self-context editing**
   - deliberate activate/deactivate/pin/unpin capability for the agent

2. **managed-state discoverability**
   - inspect/list/search/show what SCM already knows

3. **usable context-curation workflow**
   - agent can intentionally move things between metadata-only, active, and pinned roles

4. **clear SCM authority over final context**
   - SCM is not merely attached to context; SCM is what decides context

5. **agentic operating contract**
   - a normal, stable policy for how the agent should use the above capabilities
